# 执行记录 - OpenAI Responses API 转换研究与计划制定

## 1. 任务背景
在 `/Users/harvey/github/llm_proxy` 目录下设计和研究一个自研极轻量本地中转网关（TypeScript + SQLite）。该网关将底层的无状态 `/v1/chat/completions` 供应商中转并对外提供有状态、事件驱动的 `/v1/responses` 接口，解决单用户本地调试开发中的核心诉求。本次任务重点对 DeepSeek, Kimi (Moonshot AI), MiniMax, 以及 MiMo (小米大模型) 四家主流供应商进行深度兼容性分析与适配机制设计。

## 2. 执行过程记录

### 2.1 极轻量架构设计与论证
* 主代理与子代理（`generalist`）通过深度分析，确定了 **TypeScript + Bun + SQLite** 的单文件开发模式为最适合单用户调试的版型。
* 设计了 SQLite 本地单文件持久化表结构（`conversations` & `conversation_items`）和有状态拼接算法。
* 设计了 Web 网页搜索代执行二轮迭代（Agentic Loop）算法、防 TCP 粘包的 SSE 解析状态机、以及推理模型思维链（CoT）流式拦截转换状态机。

### 2.2 四家大模型供应商 API 兼容性专项研究（2026-05-25）
* 启动了 `generalist` 子代理进行这四家大模型 `/v1/chat/completions` 接口规范的深度检索与比对。
* **DeepSeek (V4 系列)**：流式思维链包含在 `choices[0].delta.reasoning_content` 中，多轮交互**必须**在 `assistant` 历史消息中带回 `reasoning_content` 字段，否则会报 400 Bad Request。支持 tool calling。
* **Kimi (`kimi-k2.6`)**：推理模型在 `reasoning_content` 中输出思维链，多轮需通过 `thinking.keep: "all"` 参数保留历史推理内容；独特的 `$web_search` 插件自执行机制（在 tools 配置并在收到 tool_calls 时回弹伪 tool 结果即可在服务端触发搜索），`kimi-k2.6` 的 thinking 模式已与 `$web_search` 兼容。
* **MiniMax**：推理模型需在请求体中指定 `"reasoning_split": true`，流式思维链包含在 `choices[0].delta.reasoning_details`（结构化数组）中，多轮需带回；不支持 completions 级内置搜索，需网关代执行。
* **MiMo (小米大模型)**：通过 `thinking` 参数开启思维链，使用 `reasoning_content` 输出并要求多轮带回；支持内置 `web_search` 插件，由控制台管理和 `force_search` 等参数控制。

### 2.3 本地网关适配方案设计
针对四家供应商设计了五大适配 and 补齐模块：
1. **会话状态与生命周期持久化**：SQLite 增加 `reasoning_content`/`reasoning_details` 存储，多轮拼接时自适应装配与强回传校验。
2. **内置工具代执行与代理循环**：对无内置搜索模型，网关拦截并执行 Tavily 触发 Agentic Loop；对支持内置搜索模型（Kimi/MiMo），网关代理配置并回弹，让搜索静默执行。
3. **思维链过滤与流式重组**：兼容 `reasoning_content` 和 MiniMax 结构化 `reasoning_details`，重组为 `response.reasoning_summary_text.delta` 事件；支持 `<think>` 标签拦截状态机。
4. **结构化输出降级**：检测底层 schema 支持度，若不支持则自动在 System 中注入 JSON Schema 约束，并出站时强行校验装配 `output_parsed`。
5. **并发排队锁**：针对 `conversation_id` 引入本地互斥锁，确保 SQLite 读写串行安全。

### 2.6 Gemini Grounding 联网搜索接地架构设计与代码重构审查（2026-05-25）
*   **Gemini 联网参数与 Grounding Metadata 深度研究**：分析了 `google_search` 与 `google_search_retrieval` 传参结构；剖析了 `webSearchQueries`、`groundingChunks` 与 `groundingSupports` 的数组关联关系。
*   **网关端核心转换器代码设计与审查优化**：
    *   **非流式反向插入算法优化**：原算法按 `startIndex` 降序在处理嵌套文本块时会发生偏移坐标失效，**子代理审查后修正为按 `endIndex` 降序**，彻底规避偏离痛点；在引用序号上增加了 `Set` 去重，防止同一 URL 多段引用出现 `[1, 1]` 标记。
    *   **流式 Transformer 架构优化**：流式 SSE 追加转换中将 `groundingChunks` 存储重构为 Map 以 URL 增量去重，**根治了数据重复推送引发的内存膨胀隐患**；在 `transform` 阶段检测到 `finishReason === "STOP"` 时直接一体化合并发送 `textDelta`、脚注与 done 信号，**规避了多次 write 产生的网络粘包**，提升了流生命周期的健壮性。
*   **双规混合智能路由设计**：提出了基于意图分析的前置路由分类器（常识实时走 Gemini 原生谷歌接地；学术源码走 Exa + MinerU 自研高精提炼管道）。

## 4. Gemini 搜索接地功能多模型专项测试（2026-05-25）

### 4.1 测试背景与目的
为了验证不同 Gemini 模型系列（特别是最新的 `gemini-3.1-flash-lite` 与 `gemini-2.5-flash-lite`）在启用了 Google Search Grounding (`google_search`) 时的实际效果，并分析其 `groundingMetadata` 返回结构，对从 `/Users/harvey/github/google_ai_aggregation/.env` 中读取的四个 Google API 密钥（`GOOGLE_API_KEY_1` 到 `GOOGLE_API_KEY_4`）进行了多模型可用性排查和搜索接地功能专项调用测试。

### 4.2 测试过程与排查发现
1. **基础连通性测试（关闭 `google_search`）**：
   - 在**不开启** `google_search` 工具的情况下，分别向 `gemini-3.1-flash-lite` 与 `gemini-2.5-flash-lite` 模型发送测试请求。
   - **测试结果**：4 个 API Key 在不带 Search 工具的请求下，对这两个模型均能成功返回 HTTP 200，并能正常生成模型文本，这证明了这 4 个 API 密钥的基础连通性与模型本身的可用性是完好且正确的。
2. **搜索接地功能调用测试（开启 `google_search`）**：
   - 在请求体中显式添加 `"tools": [{"google_search": {}}]`，提问实时气象相关问题（例如“今天北京天气怎么样？”）。
   - **测试结果表现出明显的模型系列差异**：
     - **`gemini-3.1-flash-lite`**：无论使用哪个 Key，一旦开启 `google_search`，均一致返回 **HTTP 429 RESOURCE_EXHAUSTED** 错误。
     - **`gemini-2.5-flash-lite`**：展现出不同的行为！使用 **Key 3 和 Key 4** 能够成功返回 **HTTP 200** 并正常生成带有联网搜索结果的文本，同时在 JSON 响应的 `candidates[0]` 下**成功打印出了完整的、真实的 `groundingMetadata` 结构**！而使用 Key 1 和 Key 2 依然返回 429 报错。

### 4.3 结论分析与 `groundingMetadata` 真实解构
- **差异化配额限制（Model-Specific Free Tier Limits）**：
  免费层（Free Tier）API 密钥对 `google_search` 工具的配额并非一刀切地限制为 0。配额策略不仅取决于 API Key 级别，还与调用的模型版本深度绑定。
  在当前的 API 密钥下，`gemini-3.1-flash-lite` 模型在免费层的 `google_search` 专项配额仍被限制为 0，故一致触发 429；而旧版的 `gemini-2.5-flash-lite` 模型配合特定 Key（Key 3 和 Key 4）则拥有可用的接地配额。这为我们在免费层获取真实 grounding 结构、调试解析代码提供了宝贵的真实数据源。
- **关于 `groundingMetadata` 字段结构**：
  通过 `gemini-2.5-flash-lite` 成功获取到的 `groundingMetadata` 结构包含三个关键字段，与官方 REST 规范完全一致：
  1. `webSearchQueries`（数组）：大模型在后台进行 Google 搜索的查询词（如 `["今天北京天气", "北京气温"]`）。
  2. `groundingChunks`（对象数组）：包含数据源列表。每一个 chunk 均包含 `web` 对象，其下为 `uri`（源网址）和 `title`（源标题）。
  3. `groundingSupports`（对象数组）：表示正文片段与数据源的关联。包含：
     - `segment`：具体的文本片段，带有 `startIndex` 和 `endIndex`（0 索引，字符区间左闭右开）。
     - `groundingChunkIndices`：整型数组，映射到 `groundingChunks` 里的索引，指示该片段参考的来源。

## 5. 搜索接地专项测试审查与修正（2026-05-25）

### 5.1 审查与闭环验证工作说明
于 2026-05-25 对测试脚本 `test_grounding.py` 的执行结果以及转换算法进行了专项审查，已实现如下闭环：
1. **真实数据连通性闭环**：通过 `gemini-2.5-flash-lite` 配合 Key 3/4，我们成功在真实网络环境下运行了 `test_grounding.py` 中的 `SUCCESS` 代码段（L81-L87），获取并打印出了真实的 `groundingMetadata` JSON 结构，验证了真实 API 调用的数据通路。
2. **解析与重组逻辑闭环**：
   - 审查并验证了 `test_grounding.py` 中的 Mock 解析模块，该模块模拟了网关将 API 返回的 raw 文本 and `groundingMetadata` 重组为带有引用序号的正文以及 footnotes 列表的完整逻辑。
   - **排序与插入位置算法**：算法使用 `endIndex` 降序排列，自后往前依次插入引用标记（如 `[1]`），成功避免了由于字符插入导致后续字符偏移失效的经典问题（若按升序插入，前部字符插入会不断改变后部 segment 在原字符串中的绝对索引位置）。
   - **引用标记与 footnotes 去重**：在装配时，通过判断 `fn_item` 是否已存在于 `footnotes` 中，实现了引用的去重与脚注列表的精简。即使某一段话被多次引用，在 footnotes 中也只占一个序号，且正文中标记正确（如 `[1]`）。测试输出完美实现了预期效果。
3. **网关端落地建议**：
   由于 3.1 系列模型的官方 Google Search Grounding 仍然受配额 429 限制，因此网关的双规路由设计（4.8 节）非常关键：对于 3.1 系列或配额耗尽的请求，网关应自动降级至自研的 Exa + MinerU 提炼管道；而对于支持的模型（如 2.5 系列且 Key 拥有配额时）或后续配置了 Paid Tier 的情况，则直接路由至 Google 原生 Grounding。
